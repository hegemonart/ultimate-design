# Plan 11-02: Frontmatter Feedback-Loop Proposer + Budget-Config Feedback-Loop Proposer

**Phase**: 11 — Self-Improvement  
**Requirements**: SLF-03 (frontmatter feedback loop), SLF-04 (budget-config feedback loop)  
**Depends on**: Plan 11-01 (design-reflector agent exists), Phase 10.1 (`.design/agent-metrics.json`, `.design/budget.json`)

## Goal

Extend `design-reflector` to produce two additional proposal types: (1) frontmatter updates for agents whose measured performance diverges from their declared frontmatter values, and (2) budget.json updates when telemetry shows sustained over/underspend.

Both are already handled through the `[FRONTMATTER]` and `[BUDGET]` proposal tags introduced in 11-01 — this plan adds the logic that *generates* those proposals by reading agent-metrics.json and costs.jsonl.

## Tasks

### Task 02-A — Extend `agents/design-reflector.md` with frontmatter analysis logic

Add a new section to `agents/design-reflector.md` body: **Frontmatter Analysis**.

Logic (add to the agent's "Agent performance" reflection section):

For each agent in `.design/agent-metrics.json`:
- Compare `measured_duration_seconds` (from metrics) to `typical-duration-seconds` (from agent frontmatter): if >50% deviation, emit `[FRONTMATTER]` proposal to update `typical-duration-seconds`
- Compare the agent's observed tier usage from `costs.jsonl` to its `default-tier` frontmatter: if >80% of spawns at Haiku performed acceptably (no gap-rate >10%), emit `[FRONTMATTER]` proposal to downgrade `default-tier`
- Check `parallel-safe` frontmatter: if agent-metrics show conflict events, emit `[FRONTMATTER]` proposal to downgrade from `always` to `conditional` or `never`
- Check `reads-only` frontmatter: if telemetry shows write operations from an agent declaring `reads-only: true`, emit `[FRONTMATTER]` proposal to correct

**Proposal format** (same as 11-01):
```
### Proposal N — [FRONTMATTER] Update design-X typical-duration-seconds
**Why**: measured avg 142s over 6 spawns vs declared 45s (3.2× deviation, cycle: cycle-3)
**Change**: agents/design-X.md line `typical-duration-seconds: 45` → `typical-duration-seconds: 140`
**Risk**: low
```

### Task 02-B — Extend `agents/design-reflector.md` with budget analysis logic

Add a new section to the agent body: **Budget Analysis**.

Logic for `[BUDGET]` proposals:

Read `.design/telemetry/costs.jsonl` (if exists). Aggregate per-agent:
- **Sustained overspend**: if an agent's `est_cost_usd` exceeds its budget allocation by >20% in ≥3 consecutive cycles, emit `[BUDGET]` proposal to raise cap
- **Sustained underspend**: if an agent consistently uses <40% of its budget allocation, emit `[BUDGET]` proposal to lower cap (freeing budget for other agents)
- **Consistent cap breaches**: if `cap_hit: true` appears in costs.jsonl ≥3 times for the same agent, emit `[BUDGET]` proposal to raise the cap

**Proposal format**:
```
### Proposal N — [BUDGET] Raise design-verifier per-run cap
**Why**: cap_hit in 4 of last 5 cycle runs (cycles 2–5), avg overage $0.003
**Change**: .design/budget.json → design-verifier.per_run_cap_usd: 0.02 → 0.03
**Risk**: medium
```

If `.design/budget.json` doesn't exist, skip budget analysis and note: "budget.json not found — Phase 10.1 budget governance required for budget proposals."

If `.design/telemetry/costs.jsonl` doesn't exist, skip and note: "costs.jsonl not found — Phase 10.1 telemetry required."
