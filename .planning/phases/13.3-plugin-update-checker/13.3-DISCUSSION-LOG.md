## Hot-path architecture

| Option | Description | Selected |
|--------|-------------|----------|
| Two-tier: hook writes cache + rendered nudge; agent on-demand | Hot path stays LLM-free; agent is enrichment via `--prompt` | ✓ (recommended) |
| Hook spawns agent every session | Richer summary by default, but slow + token-expensive | |
| Agent-only (skip hook, spawn on first safe-window hit per session) | Moves cost to first use; inconsistent UX | |
| Script-only, no agent at all | Skips the roadmap-named `agents/design-update-checker.md` | |

**Decision:** Two-tier. Hot path = Bash hook + `.design/update-available.md` pre-rendered. Cold path = agent only via `/gdd:check-update --prompt`.
**Notes:** SessionStart must be fast; safe-window reads must be cheap. The agent is kept (roadmap names it) but moved to an optional enrichment path.

## Hook implementation language

| Option | Description | Selected |
|--------|-------------|----------|
| Bash (matches `scripts/bootstrap.sh`) | No startup cost, no deps beyond curl; zero-dep shell semver parsing | ✓ (recommended) |
| Node (richer parsing) | Easier JSON handling, but Node startup + new dep | |
| Mixed (Bash entry + Node child for parsing) | Complexity without clear win | |

**Decision:** Bash. Roadmap literally says `hooks/update-check.sh`.
**Notes:** Shell semver parsing via awk is ~10 lines; JSON parsing needed only for the release `tag_name` + `body` fields — trivial with `grep -o` + `sed`.

## Cache + staleness

| Option | Description | Selected |
|--------|-------------|----------|
| `.design/update-cache.json` single file, 24h TTL | Fits existing `.design/` pattern; inspectable; easy to blow away | ✓ (recommended) |
| Env-var timestamp | Ephemeral across sessions — would cause repeat fetches | |
| XDG cache dir (`$XDG_CACHE_HOME`) | Cross-plugin but non-portable across OSes | |
| Split cache (one file for tag, one for changelog) | Unnecessary fragmentation | |

**Decision:** Single `.design/update-cache.json`. TTL check via `date -r` vs `date +%s` math (handle both GNU + BSD date via fallback).

## Network + API

| Option | Description | Selected |
|--------|-------------|----------|
| Unauthenticated GitHub API `/releases/latest` | 60 req/h/IP is plenty with 24h cache; no secret management | ✓ (recommended) |
| Authenticated GitHub API | Higher rate limits but needs token management | |
| Scrape releases HTML page | Fragile to GitHub UI changes | |
| npm registry fallback | Not this plugin's distribution source | |

**Decision:** Unauthenticated `/releases/latest`. `curl -sf --max-time 3`.
**Notes:** 60 req/h ÷ 24h TTL = 1440 sessions/day per IP — far beyond realistic ceiling.

## State-machine guard — which STATE.md?

| Option | Description | Selected |
|--------|-------------|----------|
| `.design/STATE.md` (plugin's design pipeline) | User-facing pipeline state — "am I mid-design?" | ✓ (recommended) |
| `.planning/STATE.md` (GSD-internal) | Wrong file; GSD is the build-the-plugin workflow | |
| Both | Unnecessary complexity | |

**Decision:** `.design/STATE.md`. Suppress when `stage:` ∈ `{plan, design, verify}`; allow in `brief`, `explore`, or when file missing.
**Notes:** Roadmap vocabulary `executing|verifying|discussing|planning` maps to `.design/` stages `design|verify|brief|plan` respectively — documented in CONTEXT.md D-12.
