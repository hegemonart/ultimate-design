# Phase 13.3: Plugin Update Checker - Context

**Gathered:** 2026-04-19
**Status:** Ready for planning
**Mode:** --auto (recommended defaults auto-selected — see DISCUSSION-LOG for rationale)

<domain>
## Phase Boundary

Ship a passive, user-respecting plugin-update checker that discovers new `get-design-done` releases out-of-band (cached GitHub Releases API poll on SessionStart), classifies the semver delta, and surfaces a one-line nudge **only in safe windows** (never mid-pipeline). Dismissal is per-version and sticky. The checker **never** auto-updates — it points the user at Phase 7's `/gdd:update` + `/gdd:reapply-patches` flow.

**In scope:**
- `hooks/update-check.sh` — SessionStart hook, 24h-cached unauthenticated `GET /repos/hegemonart/get-design-done/releases/latest`, silent-on-failure.
- `agents/design-update-checker.md` — thin enrichment agent invoked ONLY by `/gdd:check-update --prompt` for a richer summary (hot path stays script-only).
- Version-compare + semver-classify logic (major / minor / patch / off-cadence 4-segment).
- State-machine guard reading `.design/STATE.md` `stage:` — suppress nudge when stage ∈ `{plan, design, verify}`.
- Per-version dismissal flag in `.design/config.json` (`update_dismissed: "v1.0.x"`).
- Cache file `.design/update-cache.json` + rendered nudge file `.design/update-available.md`.
- Footer-banner integration in 6 safe-window skills: `/gdd:progress`, `/gdd:health`, `/gdd:help`, and post-closeout of `/gdd:ship`, `/gdd:complete-cycle`, `/gdd:audit`.
- `/gdd:check-update` manual slash command with `--refresh` / `--dismiss` / `--prompt` flags.
- Phase-13.3 baseline relock + CHANGELOG entry + `plugin.json` + `marketplace.json` bump to **v1.0.7.3** (off-cadence patch — does not shift Phase 14 → v1.0.8).

**Out of scope:**
- Auto-update. Never. The checker shows a nudge; `/gdd:update` is the user action.
- Authenticated GitHub API (unauth'd `/releases/latest` has 60 req/h/IP — sufficient for SessionStart + manual refresh at 24h cache).
- Trend/aggregate feeds (Phase 13.2's territory).
- Update nudges inside mid-pipeline skills (`/gdd:brief`, `/gdd:explore`, `/gdd:plan`, `/gdd:design`, `/gdd:verify` — explicitly blocked).
- Multi-source fallback (npm, marketplace API) — GitHub Releases is source of truth for this plugin.
- Phase 13.1 / 13.2 deliverables — independent decimal phases; only interaction is the `plugin.json` version bump (conflict-resolvable on merge).

<decisions>
## Implementation Decisions

### Hot-path architecture

- **D-01:** Two-tier design. **Hot path** = hook (Bash) writes cache + rendered nudge file; safe-window skills read the file. **Cold path** = `design-update-checker` agent is invoked ONLY by `/gdd:check-update --prompt` for a richer LLM-generated summary. **Why:** SessionStart must be fast (no LLM spawn on every session); safe-window skills must be cheap (no agent dispatch). Agent is an enrichment path, not the default.
- **D-02:** Hook implementation is **Bash** (`hooks/update-check.sh`), matching `scripts/bootstrap.sh` convention. Uses `curl -sf` + shell semver parsing (no `jq` dependency — inline awk/grep + head for changelog truncation). **Why:** roadmap explicitly names `hooks/update-check.sh`; zero new deps; matches Phase 1 cross-platform discipline.
- **D-03:** Hook registration: append a second SessionStart entry to `hooks/hooks.json` alongside the existing `bootstrap.sh` entry. Run order: bootstrap first, update-check second. **Why:** keep hooks.json structure flat; bootstrap stays load-bearing and unaffected.
- **D-04:** Silent-on-failure = literal. Hook exits 0 on any error (network timeout, JSON parse, file write, semver parse). Never prints to stderr. Never causes SessionStart to fail. **Why:** roadmap says silent-on-failure; update checker must never degrade session startup.

### Cache + staleness

- **D-05:** Cache file: `.design/update-cache.json` — single-file, `{checked_at, latest_tag, current_tag, delta, changelog_excerpt, is_newer}`. 24h TTL checked via `date -r` vs `date +%s` math in shell (cross-platform: GNU vs BSD date handled). **Why:** matches existing `.design/` artifact pattern; single file = easy to inspect/delete for testing.
- **D-06:** Rendered nudge file: `.design/update-available.md` — pre-formatted Markdown footer banner, written by hook when `is_newer=true` AND not dismissed AND stage-guard allows. Deleted when any condition fails. **Why:** safe-window skills just `cat` this file if it exists (one-line integration per skill); no re-computation needed at display time.
- **D-07:** Cache invalidation: `--refresh` flag on `/gdd:check-update` forces a fresh fetch regardless of TTL. `--dismiss` flag writes `update_dismissed: "<latest_tag>"` to `.design/config.json` and deletes `update-available.md`.

### Network + API

- **D-08:** Endpoint: `GET https://api.github.com/repos/hegemonart/get-design-done/releases/latest`. Unauthenticated. `curl -sf --max-time 3`. **Why:** 60 req/h/IP is plenty for 24h-cached SessionStart; auth adds secret-management burden with no payoff.
- **D-09:** Changelog excerpt source: GitHub Release `body` field (NOT `CHANGELOG.md` — release notes are canonical user-facing). Truncate at 500 chars with `...` suffix. **Why:** release bodies already hand-curated per `softprops/action-gh-release` from Phase 13; `CHANGELOG.md` has dev-internal detail that's noisier.
- **D-10:** Semver parsing: shell script normalizes `v1.0.7.3` → 4-segment `(1,0,7,3)`; `v1.0.7` → `(1,0,7,0)`. Lexicographic comparison per segment. Delta classification: `major` if segment[0] differs; `minor` if segment[1]; `patch` if segment[2]; `off-cadence` if only segment[3] differs. **Why:** roadmap explicitly mentions off-cadence 4-segment versions (v1.0.7.3 itself is one); lexicographic-per-segment avoids external semver tooling.

### State-machine guard

- **D-11:** Guard reads `.design/STATE.md` — NOT `.planning/STATE.md` (which is GSD-internal). Checks `stage:` field. Suppress nudge when `stage:` ∈ `{plan, design, verify}`. Allow when ∈ `{brief, explore}` or when file missing. **Why:** `.design/STATE.md` is the plugin's user-facing pipeline state; suppressing on in-flight stages protects flow; early stages (brief/explore) are discovery and can tolerate a banner.
- **D-12:** Guard mapping clarification: roadmap's nomenclature `executing|verifying|discussing|planning` maps to `.design/STATE.md` stages `design|verify|brief|plan` respectively. Internal doc note to align vocabularies.

### Per-version dismissal

- **D-13:** Dismissal scope: per-version only (sticky across sessions until a newer version ships). Stored in `.design/config.json` under `update_dismissed: "<tag>"`. If `latest_tag > update_dismissed`, nudge reappears. Dismissing via `/gdd:check-update --dismiss` is explicit; no auto-dismiss after N prompts.
- **D-14:** Config write is atomic (write-to-tmp + rename). Existing `.design/config.json` keys preserved; only `update_dismissed` touched.

### Surface windows

- **D-15:** Safe windows = 6 skills: `/gdd:progress`, `/gdd:health`, `/gdd:help`, and **post-closeout** of `/gdd:ship`, `/gdd:complete-cycle`, `/gdd:audit`. Integration = one-liner appended to each skill's final output: `[ -f .design/update-available.md ] && cat .design/update-available.md`. **Why:** roadmap explicitly lists these; closeout skills (ship/complete-cycle/audit) only show at the END of a successful run — never mid-flow.
- **D-16:** Footer banner format:
  ```
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   📦 Plugin update: v1.0.7 → v1.0.8 (minor)
   <500-char changelog excerpt>
   Install: /gdd:update   Dismiss: /gdd:check-update --dismiss
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ```
  Emoji allowed here (user-facing banner, not code/docs). **Why:** matches the visual rhythm of existing `━━━` banners in progress/health/audit output.

### Manual command

- **D-17:** `/gdd:check-update` slash command — thin skill at `skills/check-update/SKILL.md`. Flags:
  - `--refresh` — bypass 24h TTL, fetch now.
  - `--dismiss` — write dismissal flag + remove `update-available.md`.
  - `--prompt` — spawn `design-update-checker` agent for a richer summary (cold path — LLM reads release notes, generates "here's what changed for you").
  - no flag — print current cached state; if cache older than 24h, trigger `--refresh` equivalent.
- **D-18:** Agent tier: **Haiku** (per Phase 10.1 cost-governance discipline). `design-update-checker` is a summarizer/classifier — Haiku-class task. Add to `.design/budget.json` tier_overrides.

### CI/CD + release

- **D-19:** Version bump: `plugin.json` + `marketplace.json` → `1.0.7.3`. CHANGELOG.md gets a `## [1.0.7.3]` section. **Why:** roadmap explicitly names v1.0.7.3 as the off-cadence patch.
- **D-20:** Release workflow: Phase 13's `release.yml` handles tag + release automatically on push-to-main when version changes. No new CI needed. Phase-13.3 baseline relock is a single `test-fixture/baselines/phase-13.3/` snapshot.
- **D-21:** Schema additions: extend `reference/schemas/config.schema.json` with `update_dismissed: string` (optional). **Why:** Phase 13 already validates config.json via ajv; new field must pass schema to avoid validation failures.

### Claude's Discretion

- Exact wording of the footer banner (can iterate once landed).
- Exact awk/grep semver parser layout (one-liner vs small function).
- Whether `design-update-checker` agent output replaces or augments `update-available.md` when invoked via `--prompt`.
- Specific test fixture shape for mock GitHub API responses (node:test stdlib).

</decisions>

## Existing Code Insights

### Reusable Assets
- **`scripts/bootstrap.sh`** — exact pattern for a SessionStart bash hook (shebang, `$HOME` discipline, `exit 0` on all error paths).
- **`hooks/hooks.json`** — existing SessionStart registration just appends an object to the array; no new registration mechanism needed.
- **`skills/update/SKILL.md` + `skills/reapply-patches/SKILL.md`** — the exact user action the nudge points at; wording in banner should match their invocation signature.
- **`.claude-plugin/plugin.json`** — `version` field already structured for semver comparison.
- **`hooks/budget-enforcer.js`** — shows how existing hooks read `.design/config.json` safely; use the same "if-not-exist skip" pattern.
- **`scripts/validate-schemas.cjs`** — pattern for extending schemas; ensures new `update_dismissed` field doesn't trip Phase 13 CI.

### Established Patterns
- All hooks exit 0 on error (silent degradation) — never block the user.
- All repo scripts use `#!/usr/bin/env bash` + relative paths + `$HOME`. Cross-platform (Linux, macOS, Windows-in-git-bash) is non-negotiable.
- Node scripts use `node:test` + `node:assert` stdlib only.
- `.design/` is per-user-project runtime state; the plugin reads/writes it but repo has no `.design/` at rest.
- Atomic commits per task (Phase 2 discipline). A plan task touches one concern, has one commit.
- Footer-style banners use `━━━` box drawings (existing pattern in progress/health/audit output).

### Integration Points
- **SessionStart hook chain:** `bootstrap.sh` currently runs alone. 13.3 appends `update-check.sh` as a peer entry. Both must be independent-order-safe (bootstrap initializes `.design/`; update-check must gracefully handle missing `.design/` by creating it or skipping).
- **Safe-window skill surfaces:** each of the 6 skills gets a one-line tail: `[ -f .design/update-available.md ] && cat .design/update-available.md`. This is a Bash conditional in a Markdown skill file — follows the existing `Bash` tool invocation pattern in those skills.
- **`.design/config.json` schema:** already validated in Phase 13 CI. Adding `update_dismissed` requires schema bump + one passing test case.
- **Release workflow:** Phase 13's `release.yml` auto-tags on `plugin.json` version change. Bumping to 1.0.7.3 triggers release automatically; no 13.3-specific workflow needed.

<specifics>
## Specific Ideas

- **User-respect principle** (explicit in roadmap): "never interrupts critical work". This is the single most important design constraint. Every decision above flows from it — state-machine guard, safe-window list, silent-on-failure, per-version dismissal, no-auto-update.
- **Anti-slop alignment**: roadmap calls out that Phase 13.2's watcher excludes Dribbble/Behance/trending aggregation. 13.3 has a related discipline: no telemetry, no phone-home, no tracking. Unauthenticated GitHub fetch is anonymous and auditable.
- **Baseline behavior**: the nudge must be **invisible by default** for any user who stays on the latest version. A fresh install on the latest release does nothing visible after SessionStart — cache is written, no newer version exists, no nudge rendered.

</specifics>

<deferred>
## Deferred Ideas

- **Multi-plugin update aggregation** (scan all installed Claude Code plugins and show a combined nudge) — out of scope for this plugin; belongs in Claude Code CLI itself.
- **Release-notes LLM rewrite** for accessibility (e.g., "what this means for you") — the `--prompt` flag is the baseline; a more ambitious rewrite/translate path is a future enhancement.
- **Scheduled refresh outside SessionStart** (e.g., via Phase 13.2's `scheduled-tasks` MCP integration for daily fetch) — cross-phase coupling deferred until 13.2 lands.
- **Update channel selection** (stable vs beta vs pre-release) — GitHub has no beta channel for this plugin yet.
- **Rollback nudge** ("v1.0.7.3 has a known regression — downgrade to v1.0.7.2") — would require a server-side signal; out of scope.

None from scope creep — discussion stayed within phase boundary.
