## Decisions Made

- **`size_budget: M` tier cap set at 300 lines.** Lands between `S: 150` and `LARGE: 350`; accommodates CONTEXT D-05's "body ≈ 200–300 lines" target with ~10% headroom without normalizing generous budgets. No-op for existing agents (none declare `M`). Future reflector-proposed tier moves may land agents at `M` without a test change.
- **Skip-regex content boundary.** Included `sponsor(ed)?`, `newsletter`, `promo(tion)?`, `[ad]`, `subscribe`, `unsubscribe`, `webinar` (case-insensitive). These are the promo-noise signals surfaced by reviewing authority-feed seed listings in Plan 13.2-01; deliberately does NOT include "opinion" (NN/g routinely publishes opinion pieces that remain authoritative). Skip-regex is evaluated LAST so it overrides kind-based rules — a component-system release titled "Sponsored tier launch" correctly routes to skip.
- **Component-noun list is illustrative, not exhaustive.** Step 5's table spells out a sample (`button|dialog|menu|modal|card|tooltip|popover|select|combobox|etc.`) but the closing `etc.` is load-bearing: future refinements come through Phase 11 reflector proposals, not hand-edits. This respects the plan's deferred-to-reflection posture for evolving rules.
- **`named-practitioner` upgrade predicate is link-host-based**, not topic-heuristic. An entry is upgraded from `craft-tip` to `pattern-guidance` only when its link points to a spec-source host (`w3.org`, `developer.apple.com`, `m3.material.io`, `fluent2.microsoft.design`). This makes the upgrade auditable and prevents the agent from guessing at "spec-adjacency" from title text alone.
- **Fetch-notes footer is conditional.** Included only when `fetch_notes` is non-empty, so error-free runs produce a byte-identical report tail (`**Skipped:** K entries...`). This is load-bearing for Plan 13.2-04's diff-test determinism.

## Issues Encountered

- **Plan's stated baseline classification distribution overcount.** The plan (line 362) states expected baseline distribution `craft-tip: 3` and total 10 new entries. Enumerating the fixtures per the plan's own entry-by-entry specs yields: spec-change=2, heuristic-update=1, pattern-guidance=4, craft-tip=2, skip=0 → 9 entries. The plan's acceptance criteria (line 374) correctly state "Total 9 across files", so the `craft-tip: 3` figure is an informational overcount. No fixture change required — the plan's `<verify>` block and `<acceptance_criteria>` are internally consistent at 9 entries; the informational line is out of sync. Plan 13.2-04 owns the baseline file and will compute distribution against whatever fixtures ship; its diff-test reads the actual fixture output, not this plan's prose figure.
- **Pre-existing markdownlint MD060 errors in agent tables.** The repo's `npm run lint:md` was already exiting non-zero before this plan (705 errors across existing agents, mostly MD060 "table pipe alignment" noise). The new agent's classification table contains inline regexes with unescaped `|` characters inside backticks (e.g., `` `/(added|updated|new)/i` ``), which markdownlint counts as table pipes. CONTEXT D-17 and the plan's `<interfaces>` block explicitly require the table be encoded verbatim — escaping the regex pipes as HTML entities would alter the prompt content the agent reads. Scope boundary: `lint:md` is not in the plan's verify block, and the new agent contributes the same class of pre-existing lint noise. Logged here for the deferred-items list.
- **Plan's Step 4 hashing example spec vs. deterministic output.** The plan's Step 4 example uses `printf '%s' "$title\n$summary" | sha256sum`, which literally passes the two-character sequence `\n` to sha256sum rather than an actual newline. Used `printf '%s\n%s' "$title" "$summary" | shasum -a 256 | awk '{print $1}'` in the agent body instead — produces the intended `title + LF + summary` input and matches the `sha256(title + "\n" + summary)` D-13 contract. `shasum -a 256` is more portable (macOS default) than `sha256sum` (GNU coreutils only); both emit the same 64-char lowercase hex.
