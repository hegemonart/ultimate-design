**Gathered:** 2026-04-19

<domain>
## Phase Boundary

Ship `agents/design-authority-watcher.md` + `reference/authority-feeds.md` + `skills/watch-authorities/SKILL.md` that diff a curated whitelist of design-authority feeds against `.design/authority-snapshot.json`, classify new entries into one of five buckets, emit `.design/authority-report.md`, and feed that report into Phase 11's existing `design-reflector` → `/gdd:apply-reflections` pipeline without modifying the reflector agent.

**Out of scope** (belongs in future phases or explicitly rejected):
- Auto-applying reflections from authority entries — user review stays in `/gdd:apply-reflections`.
- Trend aggregators (Dribbble, Behance, LinkedIn, generic RSS farms, "trending"-style feeds). Load-bearing exclusion — see D-08.
- Paid / auth-gated feeds (Nielsen Norman Group members-only, Smart Interface Design Patterns course). Only the public RSS surfaces.
- Modifying `design-reflector.md` — reflector is input-agnostic (reads any `.design/*.md` surfaced in its required-reading). This phase produces a report the reflector can already read.
- A background daemon / polling loop. Scheduled execution is opt-in via the existing `scheduled-tasks` MCP.

<decisions>
## Implementation Decisions

### Agent shape & tier

- **D-01: Agent name** — `design-authority-watcher` (kebab-case, `design-` prefix per `agents/README.md`).
- **D-02: Default tier** — `sonnet`. Rationale: network fetch + per-entry classification is pattern-recognition work; Haiku would misclassify the spec-vs-opinion boundary; Opus is overkill for a weekly run.
- **D-03: Tools** — `Read, Write, WebFetch, Bash, Grep, Glob`. `WebFetch` is the fetch transport (no new HTTP lib). `Bash` is needed for hashing (content hash = `sha256sum` via shell) and JSON merge of the snapshot. No `Task` — watcher does not spawn subagents.
- **D-04: Parallel-safe** — `always`. Reads + writes are scoped to `.design/authority-*` files; does not race with any other agent.
- **D-05: size_budget** — `M`. Agent body ≈ 200–300 lines (classification heuristics + fetch loop + diff logic).
- **D-06: typical-duration-seconds** — `90`. Fetching ~30 feeds in parallel via a bash loop around WebFetch + classifying ~20 new entries per run.
- **D-07: reads-only** — `false`. Writes `.design/authority-snapshot.json` + `.design/authority-report.md`.

### Whitelist scope & format

- **D-08: Rejected-kinds are load-bearing** — `reference/authority-feeds.md` ships with an explicit "Rejected" section listing Dribbble, Behance, LinkedIn, "trending" aggregators, generic Medium topic feeds. Plan 13.2-04 ships a test that greps the whitelist for these hostnames and fails CI if any appear. Not a comment — an assertion.
- **D-09: Feed entry format** — markdown grouped by kind, each feed a bullet with inline fields:
  ```markdown
  - **[WAI-ARIA APG](https://www.w3.org/WAI/ARIA/apg/)** — `kind: spec-source` · `url: <rss>` · `cadence-hint: monthly` · *Normative accessibility patterns.*
  ```
  Rationale: human-editable, diff-friendly in PRs, no separate YAML file.
- **D-10: Seed feed count** — ~25–32 entries. Categories (with rough counts):
  1. Spec sources (4–5): WAI-ARIA APG, Material 3 release notes, Apple HIG (via sitemap), Fluent 2 release notes, W3C Design Tokens CG.
  2. Component systems (6–8): Radix, shadcn/ui, Polaris, Carbon, Primer, Atlassian, Ant, Mantine.
  3. Research institutions (2–3): NN/g articles RSS, Laws of UX changelog (GitHub releases), Baymard content updates.
  4. Named practitioners (10–12): curated RSS — pick writers with durable, spec-adjacent work (e.g., Adam Wathan, Ryan Mulligan, Rachel Andrew, Josh W. Comeau, Ahmad Shadeed, Sara Soueidan, Lea Verou, Scott Jehl, Jen Simmons, Heydon Pickering, Chris Coyier, Una Kravets — final list locked in plan 13.2-01 research step).
  5. User-extensible (Are.na channels): empty section with documented schema + one placeholder comment.
- **D-11: Are.na API** — use the public Are.na v2 API (`https://api.are.na/v2/channels/<slug>/contents`). No auth for public channels. Rate limit: 4 req/sec — watcher sleeps 250ms between channel fetches.

### Snapshot & diff

- **D-12: Snapshot schema** — `.design/authority-snapshot.json`:
  ```json
  {
    "version": 1,
    "generated_at": "<ISO>",
    "feeds": {
      "<feed-id>": {
        "last_fetched_at": "<ISO>",
        "etag": "<optional>",
        "entries": [ { "id": "<guid-or-url>", "hash": "<sha256(title+summary)>" } ]
      }
    }
  }
  ```
  JSON schema added to `reference/schemas/` in plan 13.2-01 so Phase 13's schema-lint picks it up automatically.
- **D-13: Diff semantics** — "new" = (new entry ID) OR (existing ID with changed content hash). Hash is `sha256(title + "\n" + summary)` — summary included because title-only updates are noise (rename).
- **D-14: Entry retention** — keep last 200 entry IDs per feed in the snapshot. Older entries are pruned on write — prevents the snapshot growing unbounded over years.
- **D-15: First-run discipline** — on the first run (no snapshot file), seed the snapshot **without surfacing anything**. Print: "Seeded snapshot for N feeds — next run will surface new entries." Escape hatch: `--since <date>` forces surfacing of entries newer than the given date on first run. Per success criterion 3 of ROADMAP.

### Classification

- **D-16: Five classification buckets** (locked by ROADMAP success criterion 3) — `heuristic-update`, `spec-change`, `pattern-guidance`, `craft-tip`, `skip`.
- **D-17: Classification rules** — encoded as a decision table in the agent body:
  | Source kind | Default classification |
  |---|---|
  | spec-source (WAI-ARIA, Material 3, HIG, Fluent 2) | `spec-change` if entry title matches `/(added|updated|deprecated|removed|new)/i` else `pattern-guidance` |
  | component-system release notes | `pattern-guidance` if title matches new-component regex, else `craft-tip` |
  | research-institution | `heuristic-update` if title/summary mentions "principle", "law", "heuristic", "usability finding"; else `craft-tip` |
  | named-practitioner | `craft-tip` by default; upgraded to `pattern-guidance` if linked to a ships-in-spec topic |
  | Are.na channel | `pattern-guidance` (user-curated references) |
  | Anything flagged as promo/newsletter/ad or matching `skip`-regex list | `skip` |
- **D-18: Classification rationale** — ≤1 sentence per entry, appended after classification (e.g., "heuristic-update: NN/g updated their error-message principle with new touch-target minimums").
- **D-19: `skip` collapse** — skipped entries never land in the body of the report. One line at the top of the report: "**N entries classified as `skip`** — see `.design/authority-snapshot.json` for the full trail if needed." This keeps the report signal-dense.

### Report format

- **D-20: Report file** — `.design/authority-report.md`. Overwritten each run. Prior runs available in git if `.design/` is committed, or in snapshot metadata otherwise.
- **D-21: Report structure**:
  ```markdown
  # Authority Report — <ISO date>

  N entries surfaced across M feeds. K skipped.

  ## spec-change (X)
  - **[Title](url)** — feed: WAI-ARIA APG — *rationale sentence*

  ## heuristic-update (X)
  - ...

  ## pattern-guidance (X)
  ## craft-tip (X)
  ```
  Sections are omitted when empty. Classification buckets ordered by weight (spec > heuristic > pattern > craft).

### Skill / command surface

- **D-22: Command** — `/gdd:watch-authorities` (skill at `skills/watch-authorities/SKILL.md`, follows naming pattern of `watch-*` — no existing collision).
- **D-23: Flags**:
  - `--refresh` — force re-seed the snapshot from current feed state without surfacing anything (recovery for a corrupted snapshot).
  - `--since <date>` — surface entries newer than the given date regardless of snapshot state (escape hatch for first run or backlog).
  - `--feed <name>` — limit to a single feed (debugging / spot-check).
  - `--schedule weekly` — only meaningful when `scheduled-tasks` MCP is connected; registers a cron. Unconnected MCP → skill prints a one-line diagnostic and exits.
- **D-24: Exit codes / completion markers** — skill terminates with `## WATCH COMPLETE` on success, or a `<blocker>` per `reference/shared-preamble.md` on fetch / parse failures (non-fatal per-feed errors are logged in the report's footer, not blockers).

### Phase 11 integration (no reflector edits)

- **D-25: Reflector read-path** — `design-reflector` already reads every `.design/*.md` listed in its required-reading block (see `agents/design-reflector.md` line 30 "minimum expected inputs"). Plan 13.2-03 appends `.design/authority-report.md` to `skills/reflect/SKILL.md`'s required-reading builder (step 3 in that skill). Zero changes to the reflector agent body itself. Contract: the report shape documented in D-21.
- **D-26: Backward compatibility** — if `.design/authority-report.md` is absent, reflector behaves as before (graceful skip per `skills/reflect/SKILL.md` step 4). This phase is additive, not replacing prior inputs.

### Testing

### Versioning + release

- **D-30: README section** — new "## Authority Watcher" section after the "Self-Improvement" section, linking to the whitelist + the command. Documents the Are.na extension point.

<specifics>
## Specific Ideas

- **Anti-slop thesis is load-bearing.** Per PROJECT.md and the plugin's Dribbble-rejection stance in ROADMAP, the authority watcher's whitelist is restricted to **authorities** — sources that ship specs, guidelines, or named-practitioner curation. The test in D-28 enforces this structurally; it is not a comment.
- **Reflector is input-agnostic.** Explicit design choice: this phase does not edit `agents/design-reflector.md`. The watcher produces a well-formed `.design/authority-report.md` that the reflector already knows how to read. Zero reflector surface area.
- **Named-practitioner list is opinionated.** The roadmap says "~12 named design writers" — the plan 13.2-01 research step locks the list with the "spec-adjacent / durable / original analysis" filter, not a follower-count metric.
- **First-run seeds silently.** Prevents wall-of-backlog. `--since` is the escape hatch.

</specifics>

- **Modifying `design-reflector`** — explicitly deferred. If reflector needs authority-specific weighting in the future, that is a separate phase after at least one cycle of real authority-report data.
